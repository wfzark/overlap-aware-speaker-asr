# AudioDepth Router Exploratory Study

## Status and Scope

Label: Frontier Branch Only / Exploratory Research.

AudioDepth is an exploratory frontier research line that currently belongs to
`frontier/audio-depth-router`. It is not a stable mainline feature, not a final
deployment claim, and not a direct merge candidate. This document preserves the
research motivation, experiment path, controlled results, negative findings,
visual explanation plan, and merge boundaries without importing the full
frontier branch or its large artifacts into `main`.

AudioDepth is best understood as a pre-ASR acoustic triage representation, not
as a production-ready standalone router. It complements transcript-instability
features; it does not replace them.

## 1. Motivation: Overlap as Time-Frequency Occlusion

The central project question is:

> When should we separate?

The answer is not "always separate" or "always use mixed ASR." Overlapping
speech creates cases where mixed ASR, separated ASR, cleaned separated
transcripts, router/cascade decisions, or human/LLM review paths can each be
reasonable. A useful system needs to choose among these paths before it has
ground-truth CER.

AudioDepth reframes overlapping speech as time-frequency occlusion. In a
spectrogram, one speaker can mask or dominate another across local time and
frequency regions, much like one object can occlude another in an image. The
router question then becomes: can a pre-ASR acoustic map expose enough overlap
structure to decide whether a text probe, separation route, cleaned transcript,
or fallback path is worth using?

## 2. Research Inspiration

The research inspiration comes from RGB-D and depth-style visual recognition.
RGB-D models do not treat depth as a replacement for RGB. They add another view
of scene geometry so a model can reason about occlusion, distance, and object
boundaries. AudioDepth applies the same spirit to multi-speaker ASR routing:
the log-mel spectrogram is the acoustic "RGB" view, while depth-like channels
encode overlap, dominance, uncertainty, or energy structure.

The analogy is useful but limited. Speech is not an image, and a depth-style
audio map is not a physical depth sensor. The goal is not to claim a perfect
visual-to-audio transfer. The goal is to create a readable acoustic
representation that exposes overlap risk before Whisper or another ASR model
has already produced an unstable transcript.

## 3. System Question: When Should We Separate?

The routing system chooses among several actions:

- mixed ASR, when the mixed transcript is likely safe and efficient;
- separated ASR, when separation is likely to recover masked speech;
- cleaned separated transcript, when duplicate or hallucinated fragments can be
  reduced;
- router / cascade, when a cheap first pass should decide whether stronger
  routes are needed;
- review / fallback, when the acoustic or transcript evidence is unsafe.

AudioDepth is aimed at the first triage step. It asks whether an acoustic map
can reduce unnecessary text probes or separation calls while blocking unsafe
direct bypass decisions.

## 4. AudioDepth Representation

Two representation modes were explored.

Deployable mode uses only mixed audio:

- channel 1: log-mel spectrogram of the mixed audio;
- channel 2: energy-based overlap proxy;
- channel 3: spectral flux, uncertainty, entropy, or related risk proxy.

Analysis-only mode can use separated tracks when available:

- channel 1: log-mel spectrogram of the mixed audio;
- channel 2: overlap activity estimated from simultaneous speaker-track energy;
- channel 3: speaker dominance based on the energy ratio between tracks.

Deployable mode is the more important long-term target because it can run before
separation. Analysis-only mode is useful for debugging and upper-bound
reasoning, but it should not be described as deployable if it depends on
separated tracks.

## 5. Visualization Design

AudioDepth maps were generated by Python code on the frontier branch to make
overlap visible as acoustic occlusion. The visualizations are explanatory
assets: they help reviewers see why a pre-ASR acoustic view might be useful.
They are not, by themselves, benchmark evidence.

The mainline repository should not absorb bulk `.png` dumps, `.npy` arrays, or
model outputs. If curated figures are added later, keep them lightweight and
review-facing. Good candidates are:

- one representative AudioDepth map;
- one mixed/separated/router decision schematic;
- one route-selection diagram;
- one safety-gate diagram.

Any future generated figure should be paired with a small reproducible script,
for example under `scripts/docs/`, and should avoid heavy dependencies.

## 6. Experiment Roadmap

### Stage A: AudioDepth MVP

The MVP built depth-augmented spectrogram maps and trained a small CNN on the
synthetic split. The purpose was to test whether the representation had any
signal for choosing `mixed`, `separated`, or `cleaned`. The result was weak:
the simple CNN did not beat router_v2. This was a negative but useful finding.
It suggested that pure local spectrogram patterns and small training data were
not enough.

### Stage B: AudioDepth Model Zoo

The next step expanded the MVP into a model zoo: log-mel-only models,
depth-augmented CNNs, handcrafted acoustic features, balanced depth models,
hybrid late-fusion models, and confidence cascades. The goal was to separate
representation weakness from model weakness. The finding was diagnostic:
handcrafted and hybrid features looked more promising than a pure AudioDepth
CNN, but the evidence remained synthetic/silver and experimental.

### Stage C: Hybrid AudioDepth + Transcript Instability

Hybrid routing combines AudioDepth with transcript-instability signals. This
matters because acoustic overlap risk and text instability describe different
failure surfaces. AudioDepth may see pre-ASR occlusion; transcript features may
see repetition, length inflation, duplicate fragments, or unstable route text.
The hybrid direction became more promising than pure acoustic routing, but it
still required careful validation.

### Stage D: Controlled Route-Sensitive Benchmark

Real-ASR slices were sometimes too route-insensitive to fairly evaluate a
router. The controlled benchmark was designed to create cases where mixed,
separated, and cleaned routes genuinely differ. Its purpose was not to create a
final deployment benchmark, but to expose route gaps and test whether a router
can make meaningful choices.

### Stage E: Balanced Benchmark v2

The first controlled benchmark was separation-dominant. Balanced Benchmark v2
added mixed-win, separated-win, cleaned-win, and review-needed targets. Current
frontier material reports a candidate pool of `240`, final benchmark size of
`120`, and `60` real Whisper evaluations. The oracle distribution was mixed
`34`, separated `26`, cleaned `0`; review-needed candidates were `57`.
Balanced route-winner router CER was reported as `0.502854`, matching oracle
CER in that slice and improving over router_v2 CER `0.643520`. This is useful
controlled evidence, but cleaned-win evidence remained limited.

### Stage F: Real Whisper Validation and Proxy-to-Real Gap

Proxy gains did not automatically transfer to real Whisper validation. Earlier
real-ASR slices had small oracle headroom, which meant even a good router had
little room to improve. This created a proxy-to-real gap audit: reference
quality, Whisper configuration, normalization, route-gap sampling, and
stratified expansion all need attention before any stable routing claim.

### Stage G: Deployable AudioDepth v2

Deployable v2 focused on mixed-only maps. Frontier notes report `200` total
maps, including `120` controlled_v2 and `80` controlled_v1, plus `100` labelled
audit samples. The best embedding probe reached oracle-route accuracy
`0.655172` and macro-F1 `0.618421`. A Stage-1 held-out gate reached accuracy
`0.758621` and macro-F1 `0.431525`, with high ambiguous/review recall but weak
easy_mixed and separation_helpful recall. This showed measurable pre-ASR
structure, but the gate was too conservative for confident routing.

### Stage H: Risk-Guarded Stage-1 Acoustic Gate

The calibrated and risk-guarded follow-ups separated route-action labels from
review/risk flags. A calibrated gate reported held-out accuracy `0.724138`,
macro-F1 `0.475000`, easy_mixed recall `0.400000`, and separation_helpful
recall `0.800000`. Its best controlled_v2 threshold sweep reported CER
`0.533160`, text-probe reduction `0.716667`, and false-safe rate `0.183333`.

The risk-guarded policy then used mixed-only uncertainty, overlap, and
overlap-uncertainty proxies to block unsafe direct bypasses. Frontier material
reports balanced controlled_v2 CER `0.529082`, route accuracy `0.833333`,
direct-bypass false-safe rate `0.000000`, and text-probe reduction `0.416667`.
These numbers belong to a controlled/frontier setting and should not be read as
broad real-meeting deployment evidence.

### Stage I: End-to-End Safety Audit

The safety audit asks whether the whole pipeline avoids high-error mixed
selections after Stage-2 fallback and review handling. The strongest
risk-guarded gate reduced unsafe direct bypasses, but frontier notes still
reported selected mixed high-error cases after fallback/review handling. That
means Stage-2 review policy remains an open problem.

## 7. Key Findings

- AudioDepth provides a plausible pre-ASR acoustic view of overlap risk.
- Simple CNN AudioDepth was weak, so the MVP is not a success claim.
- Hybrid AudioDepth plus transcript instability is more promising than pure
  acoustic routing.
- Controlled route-sensitive benchmarks are necessary because some real-ASR
  slices leave too little oracle headroom.
- Deployable mixed-only maps show measurable structure, especially for
  ambiguity/review triage.
- Risk-guarded Stage-1 gating can reduce text probes while blocking observed
  unsafe direct bypasses in controlled settings.

## 8. Negative Findings and Lessons

- The simple AudioDepth MVP did not beat router_v2.
- Pure acoustic depth models were limited.
- Proxy gains did not automatically transfer to real Whisper validation.
- Oracle headroom was small in early real-ASR slices.
- Cleaned-win evidence remained limited in balanced v2.
- AudioDepth direct-bypass safety improved, but Stage-2 fallback/review still
  caused high-error mixed selections.
- Real meeting generalization remains unproven.

These are not reasons to discard the line. They are the reason to keep it as a
frontier research program with explicit evaluation boundaries.

## 9. Visual Explanation Assets

The most useful future visual package is small and curated:

- an AudioDepth map showing overlap activity and dominance/uncertainty;

- a safety-gate diagram showing direct bypass, Stage-2 text routing, and review.

Do not merge bulk generated images from the frontier branch. If a figure is
added, include only one to three lightweight `.png` or `.svg` assets and the
small script needed to reproduce them.

## 11. What Must Stay Out of Main

- model weights;
- large `.npy` files;
- bulk generated `.png` depth maps;
- raw exploratory result dumps;
- duplicated wave/writeback records;
- unverified benchmark claims;
- direct merge of `frontier/audio-depth-router`.

## 12. Limitations

The strongest numbers above are controlled/frontier evidence. Some references
are `silver_plus_unverified`, not gold. The frontier branch includes artifacts
that need storage review. Real meeting generalization is not proven. The
current deployable gate still needs stronger false-safe control, Stage-2
fallback policy, and real-ASR validation on route-sensitive slices.

## 13. Suggested Next Steps

1. Keep this document as the mainline overview.
2. Extract only lightweight diagrams or examples if reviewers need visuals.
3. Split any future merge into separate PRs for docs, figures, minimal code, and
   tests.
4. Expand real Whisper validation with route-sensitive sampling.
5. Audit reference quality before claiming benchmark-level improvements.
6. Keep AudioDepth labelled Frontier Branch Only until the above evidence is
   reviewed.
